## Code architecture

The foundation is built. **`src/pages/dog-boarding-school.astro` is the reference implementation** for every later page ŌĆö mirror its conventions rather than inventing new ones.

**Page model.** Every page is a `<Base title description path ogImage? ogType?>` wrapper. `src/layouts/Base.astro` owns the whole `<head>` (title/description/`canonical`/OG/Twitter, with `canonical = business.domain + path`; `ogType` defaults to `'website'` ŌĆö blog posts pass `"article"`), the **site-wide JSON-LD `@graph`** (see Schema below), Consent Mode v2 (defaults **denied** in an inline script *before* GTM loads ŌĆö UK PECR), the GTM container (`business.tracking.gtm`), Astro `<ClientRouter />` (view transitions), and the `Header` / `<slot/>` / `Footer` / `ConsentBanner` shell. Pages supply only their `<main>` content + the `<Base>` props.
- **`updated` prop (optional, added 2026-07-18 ŌĆö GEO freshness).** Pass an ISO `YYYY-MM-DD` and Base renders a visible *"This page was last reviewed and updated &lt;Month Year&gt;"* line at the foot of `<main>` **and** emits a `WebPage` node carrying `dateModified` (referencing `WEBSITE_ID`/`ORG_ID` by `@id`, never restating them). Freshness is a real ranking + AI-citation signal (the <3-months cohort is cited ~3├Ś more), **but only if the date is true: bump it when the content genuinely changes, never on a rebuild, and NEVER wire it to the build clock** ŌĆö that would tell the same lie on every page at once. **Analytics ŌĆö GA4 lives in GTM, and `src/scripts/analytics.ts` exists for ONE reason** (built 2026-07-14; before this the site had **never collected a single analytics event**). Property **"The Fairy Tails ŌĆö Website" `G-TPBSKV29CJ`** (UK tz, GBP, 14-month retention). - **­¤ö┤ THE `<ClientRouter />` TRAP.** Navigation is client-side, so GTM's Google Tag fires only on a **hard load**, and GA4's Enhanced Measurement option *"page changes based on browser history events"* is **deliberately DISABLED on the data stream** (it fires mid-swap and records the **previous** page's `<title>`). A stock GA4 install therefore sends `page_view` **once** and then goes silent ŌĆö it *looks* installed while under-reporting massively. Client-side navigations are reported by `analytics.ts` on **`astro:page-load`**, the only moment the new DOM (and `<title>`) is live; the **first** `astro:page-load` is skipped because the hard load was already counted. Verified live: hard load ŌåÆ 0 extra pushes; view transition ŌåÆ exactly 1, carrying the NEW title.

**Animation contract** (GSAP + ScrollTrigger in a per-page `<script>`):
- Register the plugin; run an **idempotent** `init()` guarded by the standard `data-gsapInit` flag (`if (!root || root.dataset.gsapInit) return;`, then set `root.dataset.gsapInit = '1'` ŌĆö reuse this exact flag, don't invent a per-page variant); wrap motion in `gsap.matchMedia('(prefers-reduced-motion: no-preference)')`. **Reduced-motion returns early ŌåÆ the page is fully visible and static.**
- **­¤ö┤ A SCROLL REVEAL MAY MOVE CONTENT. IT MUST NEVER HIDE IT.** Reveals are **y-only** (`gsap.from(el, { y: 36 })`) ŌĆö no `autoAlpha: 0`, no `opacity: 0`, ever, on anything carrying text. `gsap.from()` **immediate-renders** its from-state, so a hidden from-state hides the element the instant JS runs and only un-hides on scroll. **Googlebot does not scroll, and it snapshots before GSAP's timeline advances** ŌĆö so it rendered `/dog-boarding-school` with **all 14,227 characters of body copy `visibility:hidden`**, the long tail collapsed, and clicks fell **71%** (2026-07-14; full write-up in `SEO.md`). The old rule here ("set hidden states in JS, never CSS, so a no-JS page still shows content") was **wrong**: it protected the no-JS case but not the JS-runs-but-never-animates case, which is every crawler and every stalled render. Content must be fully visible in the rendered DOM at **t=0**, with no JS and no scroll required. `npm run verify-reveals` is the gate ŌĆö it fails the build on any `gsap.from()` that hides content. Movement (`y`/`x`/`scale`/`blur`) is fine; only opacity/visibility are banned. **Decorative, aria-hidden furniture may still fade** (the `[data-hero-cue]` scroll arrow, the homepage canvas/video crossfade) ŌĆö those are `.to()` tweens carrying no text, and the gate ignores them. Standard hooks: `data-reveal` (slide-up on scroll), `data-stagger` (stagger a group's children), `data-hero-*` (hero entrance timeline).
- **Hero offsets must not cross.** In a y-only stagger, an element may not start displaced *more* than the one above it, or it visibly slides over its neighbour (the fade used to mask this). The hero title was `y: 54` against a `y: 24` sub-line with a 24px gap ŌåÆ the H1 sat *through* the sub-line on first paint. All hero elements now use `y: 24` (CTA 18, trust 36 ŌĆö both below everything they could collide with).
- **If a CSS rule must own an element's opacity afterwards (e.g. a dim/spotlight state), don't tween `autoAlpha`/`opacity` on it** ŌĆö a `gsap.from(..., {autoAlpha: 0})` leaves inline `opacity: 1` behind that silently defeats the CSS forever (2026-07-05 review lesson; the puppy page's class-night steps use an x-only entrance for this reason ŌĆö the day/regime-step dim rules on Day School / Intensive / B&T still carry the no-op, decide at polish).
